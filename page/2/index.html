get和post请求

Posted on 2015-06-22

get和post请求

http协议（超文本传输协议，基于请求和响应的协议）

所谓协议：规定、规范，由W3C起草
作用：用来规范浏览器与web服务器进行数据交互的一种格式
web服务器不管是客户端是IE、火狐还是谷歌，只要都按照http协议就可以解析
可以传输文本、图片、视频

http协议的数据格式分为：请求数据格式、响应数据格式

发送请求

  • get方式

    • 地址栏
    • 点击超链接
    • 直接在地址后面加需要的参数名称，参数对应的值
    • 多个参数时，用&连接
    • aaa

  • post方式

    • 表单提交

区别：

  • get方式没有请求体
  • get请求要给服务器发参数都在路径的后面加，大小受到限制
  • post方式有请求体，内容就是发给服务器的参数，数据大小没有限制
  • get提交方式会乱码，用post则不会，因为浏览器把请求体转码了

请求首行——告诉服务器时get/post请求，告诉服务器要请求的内容地址、协议版本
请求头——每个请求头都有特定含义
请求空行——分隔的空行
请求体——发送到服务器的数据参数，只有post有请求体

refer——请求头中的refer告诉服务器，当前请求来自哪个页面，地址栏输入时没有refer，超链接和表单提交有refer
user-Agent——告诉服务器浏览器版本

响应首行——包含http协议版本
状态码——200：响应成功
状态码——404：请求资源未找到（路径出错）
状态码——500：服务器内部出错，有bug
响应头—— Content-Type：text/html
告诉浏览器响应的文件格式
Refresh：4，url=http：//www.baidu.com
4秒后跳转到百度
响应空行——分隔的空行
响应体——html页面

关于bower的使用

使用 bower 管理库

bower 客户端库管理工具

bower 是一个适用于 web 端第三方包的自动化解决方案

  • 安装包
    • 自动安装依赖项
  • 卸载包
  • 更新包
  • 查找包
  • 保存依赖项

bower 的安装

  • 基于 node 环境
  • 基于 npm 包管理工具
    • 基于 node 开发的一个包管理工具
    • npm -v
    • npm 在安装 node 时就被自动安装了
  • 基于 git 环境
  • npm install –global bower
    • 在任意目录都可以直接执行
    • 安装 bower ，其实就是 npm 下的一个分项
    • 在全局安装一个命令行软件（cli）
    • 安装结束后通过 bower –version 查看版本，确认是否安装成功
    • 或者 工具名 –help 可以查看帮助

bower 基本使用

  • bower install 包名
    • 在哪个项目下执行，这个包就安装到哪个项目根路径
    • 当下载成功时，默认安到项目根路径下的 bower_components 中
    • 这些包都是从 github 上下载的最新稳定版本
  • bower uninstall 包名
    • 卸载包
  • bower init
    • 保存依赖
    • 在目录下生成 bower.json
    • 记录该项目所依赖的所有包
    • 即便不小心删除了 bower_components ，也可以通过 bower.json 恢复
  • bower install
    • 只要 bower.json 存在，即可重新全部安装保存的依赖包
    • 上传到 github 时，不需要传包，只要上传 bower.json
    • 别人要下载时，先 bower install 安装依赖
  • bower install –save
    • 将安装的包添加到 bower.json 中的依赖项中
    • 不加 –save 的话，下载包之后 bower.json 中没有记录
  • .bowerrc 文件
    • 该文件必须放到项目根路径下，即执行 bower 命令所处的文件目录
    • 可以用来配置 bower 的默认行为，例如可以修改安装包的默认路径
  • bower install 包名 –offline
    • 离线下载
    • bower 对下载过的包有缓存记录，可以离线下载

bower.json 文件中的 dependencies

dependencies字段指定了项目运行所依赖的模块， 它们都指向一个对象。
该对象的各个成员，分别由模块名和对应的版本要求组成，表示依赖的模块及其版本范围

npm

一个包管理工具：node 包和前端包，他包含了 bower 的所有功能

什么是跨域

同源策略，它是由Netscape提出的一个著名的安全策略。

现在所有支持JavaScript 的浏览器都会使用这个策略。

所谓同源是指，域名，协议，端口相同。

当一个浏览器的两个tab页中分别打开来 百度和谷歌的页面

当浏览器的百度tab页执行一个脚本的时候会检查这个脚本是属于哪个页面的，

即检查是否同源，只有和百度同源的脚本才会被执行。

XMLHttpRequest 有跨域请求限制

在地址栏中直接输入则不会有跨域

  • 协议名 protocol
  • 主机名 host
  • 端口号 protocol

这三个有任意一个不同，网站建的数据请求与传输便构成了跨域

不同域名、主机名会有跨域限制
127.0.0.1
192.168.123.123

同域名不同端口号hiU有跨域限制
127.0.0.1：3000
127.0.0.1：5000

不同协议会有跨域
file：//c:abc
http://www.abc.com
https://www.abc.com

跨域的解决方案

  • jsonp

img 标签没有跨域限制，但是得不到响应内容
link 标签没有跨域限制，能拿到响应内容
a 标签有跨域限制
所有能指定 src href 的标签都没有跨域限制
script 标签没有跨域限制

当给 script 标签指定了 src 属性后，浏览器会对该地址发送请求
当请求响应成功之后，也就是拿到响应内容之后
浏览器会把得到的响应内容当作 JacaScript 语法去解析
所以把相应内容写成 JS 语法
这时候，服务器的响应换成了 js 语言
若响应内容为 json 格式 {“a”:”b”}
则拿不到，若内容为 var obj={“a”:”b”}
则能拿到数据

  1. 创建一个 script 标签
  2. 给 script 标签指定 src 属性
  3. 将 script 标签放到 body 中

var scriptElement = document.createElement(‘script’)
scriptElement.src=’http://127.0.0.1‘
// 当把 script 标签放到 DOM 中之后，并不是马上发起请求的
// 而且 script 指定的请求什么时候成功无法确定
// 也就是说响应回来的代码什么时候执行无法确定
document.body.appendChild(scriptElement)
// 当该标签请求成功并解析执行完 JS 代码之后，触发自己的 load 事件
// 然后执行该回调函数
// 思考：我们能不能在发送请求的时候，就告诉对方，给我定义一个什么变量名
// 让对方查询字符串，定义该变量名的对象
// 那么我们在使用的时候就能直接拿这个变量名来使用
scriptElement.addEventListener(‘load’,function(){console.log})
// 这样通过在后台获取到请求的变量名，将数据对象命名为该变量名
// 通过这种方法，我们前端就可以定义数据的变量名

// 但是该方法不太好，升级一下
scriptElement.src=’http://127.0.0.1:3000?callback=getData‘
后台响应 queryObj.callback +’（’+数据内容+’）’
function getData(data){
得到数据内容 data
}

现在封装一下这个方法，因为前后台交互常采用 json 格式
所以叫 jsonp ， p 是 pack 包装的意思

function jsonp(url,{params},fn){
//第一步，解析处理 url
var queryString = ‘’
for(var key in params){
queryString+=key + ‘ = ‘ + params[key] + ‘&’
//结尾多一个& 不熟影响
}

// 第二步，创建 script 标签，设置 src ，添加到 body
var scriptElement = document.createElement('script')
scriptElement.src=url
document.body.appendChild(scriptElement)
// 将用户传递进来的函数 fn 变为全局函数 fn
window[abc] = fn
//此时还有问题,每次暴露出去的函数名都是全局变量，多次调用jsonp会覆盖掉函数
var callbackName = 'abc' + (Math.random()).toString() 
window[callbackName] = fn
url = url + '?' + queryString + 'callback ' + callbackName'

}

使用 Hexo 搭建自己的个人静态博客

  • 必须安装 Node 环境
    • 通过在终端任意目录输入 node -v 查看是否有输入node的版本
  • 在终端的任意目录输入 npm install -g hexo-cli 然后回车执行等待安装成功
    • 通过输入 hexo –version 如果看到输入一大堆的版本号就说明安装成功了
    • Hexo 是一个命令行应用程序
  • 接下来通过输入 hexo init 博客项目存储目录
    • Hexo 会在你执行该命令的目录下生成一个博客项目
    • 初始化博客项目的过程需要联网
  • 接下来通过终端进入刚才创建的博客项目目录，然后在该目录下输入 npm install 回车等待执行结束
  • 接下来输入 hexo server 回车执行，它会在你的本地启动一个服务器，然后就可以访问你的博客项目
  • 进入博客项目，输入 hexo new 文章名称 创建博客
    • 该命令会自动在 source/_posts/ 目录下生成一个 文章名称.md 文件
  • 生成静态文件：hexo generate
    • 该命令会自动在项目的根目录下的 public 目录下降静态文件放进去
  • 快速部署
    • 修改配置文件
    • 在项目根路径下执行 npm install hexo-deployer-git –save 该命令，为 hexo 安装 git 插件
    • 接下来就可以直接输入 hexo deploy –generate 自动发布到 github 上
  • 评论功能
    • 通过安装第三方插件，国内使用最多的是“多说”
    • 使用 多说 for node.js
    • 因为 hexo 是通过 node.js 开发的
